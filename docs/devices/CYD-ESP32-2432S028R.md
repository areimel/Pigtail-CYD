# Cheap Yellow Display — ESP32-2432S028R (Target Device)

Reference for the board **Pigtail-CYD** targets. The "Cheap Yellow Display" (CYD) is a
family of low-cost integrated ESP32 + TFT touchscreen boards (yellow PCB) made primarily
by Sunton/JINGCAI, popularized by Brian Lough's
[ESP32-Cheap-Yellow-Display](https://github.com/witnessmenow/ESP32-Cheap-Yellow-Display)
repo. We target the canonical **ESP32-2432S028R** (2.8").

> ⚠️ **Chip downgrade vs the Cardputer.** The Cardputer is an ESP32‑**S3**; the base CYD is the
> *original* ESP32‑WROOM‑32 (Xtensa LX6). No native USB, no PSRAM, less RAM. This drives most of
> the porting constraints — see [RAM budget](#ram-budget) and [USB / flashing](#usb--flashing).

## At a glance

| Property | Value |
|---|---|
| Module | ESP32-WROOM-32 (Xtensa dual-core LX6 @ 240 MHz) |
| Flash | 4 MB |
| PSRAM | **None** |
| SRAM | 520 KB total (~290–300 KB free heap after Wi-Fi up) |
| Wireless | Wi-Fi 802.11 b/g/n (2.4 GHz) + Bluetooth 4.2 (Classic + BLE) |
| Display | 2.8" TFT, 240×320 portrait (**320×240 landscape**), SPI |
| Display driver | **ILI9341 _or_ ST7789** (varies by batch — see [variants](#display-driver-lottery)) |
| Touch | XPT2046 **resistive**, on a separate SPI pin set |
| USB | micro-USB (some "R2/CYD2USB" revisions add a flawed USB-C) |
| USB-serial bridge | CH340 or CP2102 (**no native USB-CDC**) |
| Onboard extras | RGB LED, LDR light sensor, small speaker/amp, microSD slot |
| Price | ~$10–15 |

## Pinout

### Display (SPI TFT)
| Signal | GPIO |
|---|---|
| SCLK | 14 |
| MOSI | 13 |
| MISO | 12 |
| CS | 15 |
| DC | 2 |
| RST | not on a GPIO (tied to reset) |
| Backlight (BL) | 21 (PWM) |

### Touch (XPT2046, resistive) — **separate SPI pins**
| Signal | GPIO |
|---|---|
| CLK | 25 |
| MOSI | 32 |
| MISO | 39 |
| CS | 33 |
| IRQ | 36 (input-only) |

### microSD (VSPI)
| Signal | GPIO |
|---|---|
| CS | 5 |
| MOSI | 23 |
| MISO | 19 |
| SCK | 18 |

### Onboard peripherals
| Peripheral | GPIO | Notes |
|---|---|---|
| RGB LED — Red | 4 | **Active LOW**, common anode |
| RGB LED — Green | 16 | Active LOW |
| RGB LED — Blue | 17 | Active LOW |
| LDR light sensor | 34 | ADC1, input-only |
| Speaker / amp | 26 | drive with PWM/LEDC tone |
| BOOT button | 0 | also a strapping pin |

### Free / exposed GPIO (very limited)
Only a handful are truly free on the P3/CN1/P1 headers: **GPIO 22, 27** (general purpose) and
**GPIO 35** (input-only, ADC). GPIO 21 can be repurposed if backlight control isn't needed.
A future optional external GPS UART would live here.

## Known quirks & gotchas

### Display driver lottery
Boards sold under the **same** "ESP32-2432S028R" name ship with **either an ILI9341 or an
ST7789** controller; newer batches increasingly use ST7789. There is no reliable way to know
before powering on — check the IC marking on the display flex, or try one driver and fall back
to the other. Pigtail-CYD exposes a compile-time switch (`-DCYD_PANEL_ST7789`, default ILI9341).

### Color inversion / BGR order
Many boards (especially ST7789 variants and the USB-C "R2" revision) show **inverted or
BGR-swapped colors**. Fixes: `invertDisplay(true)` and/or setting the panel RGB order to BGR.
In LovyanGFX this is `cfg.invert` and `cfg.rgb_order` on the panel config. The R2/CYD2USB
variant also has gamma quirks.

### Touch + SD shared-SPI conflict
The XPT2046 touch controller and the SD card are wired such that naive setups conflict on the
hardware SPI buses. The robust solution is to drive touch via software/bit-bang
(e.g. `XPT2046_Bitbang_Slim`) or, as Pigtail-CYD does, configure touch as a panel inside the
LovyanGFX device and keep SD on its own VSPI pins — **verify coexistence on hardware** (Phase 5).

### RAM budget
On the no-PSRAM WROOM (~520 KB SRAM, ~290 KB free heap after Wi-Fi), the simultaneous load is
tight. Approximate Pigtail-CYD footprint:

| Consumer | Approx. RAM |
|---|---|
| Wi-Fi stack | ~50 KB |
| NimBLE stack | ~30–40 KB |
| 320×240 **4bpp indexed** sprite framebuffer | **~38 KB** |
| Device/anchor state arrays (`g_tracks`, `g_anchors`, UI `_items`) | ~80–100 KB |

The **4bpp indexed sprite is the key enabler** — a 16-bit framebuffer would be ~150 KB and would
not fit alongside Wi-Fi + BLE. The vendor table (`MacPrefixes.h`), icons, logo, and palettes live
in **flash**, not RAM. Phase 3 of the build measures real free-heap headroom and tunes
`MAX_TRACKS` / `MAX_ANCHORS` / UI `_items[]` down if needed.

### USB / flashing
No native USB-OTG/CDC (it's the original ESP32). USB goes through a **CH340 or CP2102** UART
bridge, appearing as `COMx` / `/dev/ttyUSB*` at **115200 baud**. Windows often needs the **CH340
driver** installed manually. Flash with esptool/PlatformIO; hold **BOOT (GPIO0)** if the board
doesn't auto-enter the bootloader. The micro-USB port is standard; the "R2/CYD2USB" USB-C port
lacks CC resistors and won't work with USB-C-to-USB-C cables.

## How Pigtail-CYD maps to this hardware

| Function | CYD mechanism |
|---|---|
| Display | LovyanGFX `LGFX_Device` for ILI9341/ST7789 on the TFT SPI pins above (`src/Display.*`) |
| Input | XPT2046 touch via `lcd.getTouch()` + on-screen action bar (replaces Cardputer keyboard) |
| Audio | LEDC tone on GPIO26 (replaces `M5Cardputer.Speaker`) |
| Battery | **removed** — CYD has no fuel gauge |
| Storage | microSD on VSPI (KML export) + SPIFFS (watchlist/ignorelist JSON) |
| GPS | **removed** — segmentation falls back to Wi-Fi AP fingerprint |
| Wi-Fi sense | `esp_wifi` promiscuous mode (unchanged, generic ESP32) |
| BLE sense | NimBLE scan (unchanged, generic ESP32) |

## CYD family (for future reference, not v1 targets)

| Model | Chip | Display | Driver | Touch |
|---|---|---|---|---|
| ESP32-2432S024 | WROOM-32 | 2.4" 240×320 | ILI9341 | XPT2046 |
| **ESP32-2432S028R** | **WROOM-32** | **2.8" 240×320** | **ILI9341/ST7789** | **XPT2046** |
| ESP32-3248S035 | WROOM-32 | 3.5" 480×320 | ST7796 | XPT2046 / GT911 |
| ESP32-8048S043/050/070 | **S3**-WROOM-1 (8 MB PSRAM) | 4.3–7" 800×480 | ST7262 (parallel RGB) | GT911 capacitive |

The S3-based 8048 series removes the RAM constraint and the SPI touch conflict but uses a
fundamentally different (parallel RGB) display path — a candidate for a future port.
